DELIVERABLE 1: ENSEMBLEPREDICTOR MODULE
================================================================================

Location: 'fy'-suites/fy_platform/ai/ensemble_predictor.py
Lines: 280 (within 300 limit)

PURPOSE:
Combine Phase 3 baseline cost models with Phase 4 historical learning to
provide ensemble predictions with variance estimates and confidence intervals.

KEY METHODS:
  - ensemble_cost_estimate(suite_list) → EnsemblePrediction
  - ensemble_criticality(suite_list) → float (0.0-1.0)
  - prediction_variance(suite_list) → float
  - confidence_interval(suite_list) → tuple (lower_bound, upper_bound)
  - variance_reduction_vs_phase3() → float (%)
  - meets_variance_target(target_pct) → bool

ALGORITHM:
  1. Get Phase 3 baseline estimate: cost_estimate = CostModelBuilder
  2. Get Phase 4 refined estimate: refined_cost = HistoricalCostAnalyzer
  3. Calculate confidence weights based on data quality
  4. Weighted average: ensemble_cost = (phase3_weight × cost) + (phase4_weight × refined)
  5. Calculate variance from model disagreement
  6. Generate 95% confidence intervals using normal distribution
  7. Cache predictions for performance

VARIANCE REDUCTION:
  Target: ≥15% variance reduction vs Phase 3 alone
  Achievement: Conservative 5%+ reduction (realistic with small datasets)
  Method: Ensemble averaging reduces prediction variance through model diversity

CONFIDENCE INTERVALS:
  - 95% confidence interval using 1.96 × standard deviation
  - Bounds: [ensemble_cost - margin, ensemble_cost + margin]
  - Margin grows with prediction uncertainty

DELIVERABLE 2: ANOMALYDETECTOR MODULE
================================================================================

Location: 'fy'-suites/fy_platform/ai/anomaly_detector.py
Lines: 270 (within 300 limit)

PURPOSE:
Detect outcome outliers, regime shifts, broken fixtures, and suite updates
using pure statistical methods (no machine learning).

ANOMALY TYPES DETECTED:
  1. outlier (Z-score > 2.0): Cost/criticality divergence
  2. regime_shift: Distribution change in recent vs historical data
  3. fixture_broken: Fixture failure rate > 50%
  4. suite_update: Performance change after suite modifications

KEY METHODS:
  - detect_anomalies() → list[AnomalyRecord] (all anomalies)
  - is_anomalous_outcome(outcome) → bool (single check)
  - regime_shift_detected() → bool (any shift detected)
  - anomaly_severity(outcome) → float (0.0-1.0)
  - anomalies_by_type() → dict (grouped by type)

DETECTION ALGORITHMS:

  Z-Score Outlier Detection:
    1. Calculate mean and std dev of outcomes
    2. For each outcome: z_score = |actual_cost - mean| / std_dev
    3. Flag if z_score > 2.0 (99.7% confidence)
    4. Severity = (z_score - threshold) / 3.0

  Regime Shift Detection:
    1. Split outcomes into early (old) and recent (new)
    2. Calculate mean cost for each period
    3. Shift magnitude = |recent_mean - early_mean|
    4. Shift score = magnitude / early_std_dev
    5. Flag if score > 0.5 (0.5 std dev shift threshold)
    6. Severity = shift_score / 1.5

  Broken Fixture Detection:
    1. Group outcomes by fixture
    2. Calculate failure rate per fixture
    3. Flag if failure_rate > 50%
    4. Severity = failure_rate

REGIME SHIFT TARGET:
  Target: Catch ≥90% of known regime shifts
  Achievement: Split analysis captures mean changes
  Threshold: 0.5 std dev shift = significant regime change

Feature 3a: DATABASE RETRY LOGIC
  Location: outcome_persistence.py::retry_on_db_error()

  Implementation:
    - Max retries: 3 (configurable)
    - Exponential backoff: 0.1s, 0.2s, 0.4s (2^attempt × 0.1s)
    - Handles: sqlite3.OperationalError, sqlite3.DatabaseError

  Usage:

    outcome_id = persistence.store_outcome(outcome)  # Auto-retries

Feature 3b: OUTCOME DEDUPLICATION
  Location: outcome_persistence.py::check_duplicate_outcome()

  Implementation:
    - Checks: same suites AND similar costs (< 0.01 USD difference)
    - Prevents storing identical compositions twice
    - Enabled by default in store_outcome()

  Usage:
    outcome_id = persistence.store_outcome(outcome, check_duplicate=True)

Feature 3c: LEARNING RATE CAPPING
  Location: criticality_learner.py::learning_rate_cap()

  Implementation:
    - Max refinement: 5% per outcome (configurable)
    - Prevents overtraining on single noisy outcomes
    - Maintains stability in criticality learning

  Formula:
    max_change = max_learning_rate × current_criticality
    if |new_signal - current| > max_change:
      capped_signal = current + sign(change) × max_change
    else:
      capped_signal = new_signal
