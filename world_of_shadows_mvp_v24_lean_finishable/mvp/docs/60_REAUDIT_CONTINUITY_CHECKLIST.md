# 60 — Re-Audit Continuity Checklist

Use this checklist immediately after an implementation AI returns work.

## A. Scope alignment
- Did the returned work target the requested field?
- Did it stay inside the instructed repository surfaces or package surfaces?
- Did it avoid unrelated redesign?

## B. Completeness check
- Were the requested behaviors actually implemented?
- Were constraints operationalized instead of restated?
- Were acceptance expectations addressed?

## C. Evidence check
- Were tests added or improved?
- Do the tests exercise the claimed behavior?
- Was validation output reported honestly?

## D. Runtime readiness check
- Did the change improve real runtime capability?
- Did it strengthen authority, validation, memory continuity, or observability?
- Did it reduce a blocker rather than just add more design material?

## E. Drift check
- Did the returned work violate authoritative world-engine law?
- Did it blur player/operator separation?
- Did it overclaim production readiness?

## F. Delta note
Record:
- improved,
- unchanged,
- partial,
- regressed,
- or drifted.

## G. Next-cycle decision
Choose one:
1. continue the same field,
2. harden evidence for the same field,
3. or move to the next highest-value field.
