# 00 — Master MVP

## Purpose

This MVP defines the full World of Shadows target as an implementable system, not as a concept bundle.
It combines:
- the constitutional laws,
- the authoritative runtime architecture,
- the player and operator surface model,
- the memory-domain runtime,
- repository ownership rules,
- executable proof code,
- and the validation path required to turn the MVP into finished implementation.

## Core position

World of Shadows is an authoritative narrative runtime.

1. The world-engine is the only live truth boundary.
2. AI can propose, but only validated commits become canonical truth.
3. Player-visible output must derive from committed or lawfully staged surfaces.
4. Runtime memory is multi-domain, temporal, conflict-aware, threshold-aware, carrier-aware, and effect-aware.
5. Emotional and consciousness layers are runtime participants, not decorative annotations.
6. The MVP itself must show where implementation belongs and how it is validated.

## What this MVP contains

This MVP now includes three inseparable layers:

### Layer A — Canonical target definition
The product, runtime, memory, governance, and surface chapters define what World of Shadows is.

### Layer B — Implementation guidance inside the MVP
The later chapters define exactly how the target maps onto the real repository, which components own which responsibilities, how audit findings translate into code work, and how work is accepted.

### Layer C — Executable proof slice
`reference_scaffold/` provides a runnable minimum slice that proves the constitutional runtime shape in working code.

## Acceptance stance

A future PASS requires all of the following:
- executable seams in the owning component,
- matching tests,
- validation evidence,
- explicit degraded behavior,
- and release honesty about what is proven versus what still requires implementation.

## Package rule

This MVP must be usable on its own.
Nothing essential to understanding the implementation path should exist only as an external note or afterthought.
